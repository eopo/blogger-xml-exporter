# Blogger XML Exporter

A full-stack application to export Blogger posts as custom XML documents. Select a blog post with full-text search, complete a configurable form, and generate XML output.

**Stack**: Vue 3 + TypeScript (frontend) | Go 1.26 (backend) | Tailwind CSS

## Prerequisites

- **Node.js** 20+ (frontend development)
- **Go** 1.26+ (backend)
- **make** (build commands)
- **Blogger API Key** (see [Setup](#setup))

## Quick Start

### 1. Setup

```bash
git clone <repository-url>
cd blogger-xml-exporter

# Copy and configure
cp config.example.yaml config.yaml
# Edit config.yaml as needed

# Install dependencies
make install
```

### 2. Development

```bash
# Start Vite dev server (port 5173) + Go backend (port 8080)
make dev
```

Open `http://localhost:5173` in your browser.

### 3. Build

```bash
make build
```

Output:
- Frontend: `web/static/`
- Backend: `bin/blogger-xml-exporter`

### 4. Run

```bash
export BLOGGER_API_KEY="your-api-key"
./bin/blogger-xml-exporter
```

Server runs on `http://localhost:8080`.

### `config.yaml` Structure

**blogger**: Blog ID and post fetch limit

**server**: Server port

**site**: Application title and heading

**assets**: Optional favicon/logo directory (Docker volume mount)

**theme**: Optional color scheme (primary, dark, light)

**form**: Form definition tree
- **type**: "group" (sections) or field type (text, textarea, date, select, array, etc.)
- **source**: Dot-path into Blogger post JSON for auto-fill
- **template**: Go text/template for computed values (supports `source`, `stripHTML`, `unescapeUnicode`, regex, etc.)
- **row/width**: Grid layout (12 columns); items with same row number appear inline
- **presets**: Quick-fill templates for groups

**xml**: XML output definition
- **root**: Root element name
- **filename**: Template for output filename (without `.xml`)

See `config.example.yaml` for complete reference.

## Project Structure

The project is organized as a monorepo with clear separation of concerns:

**`backend/`** – Go HTTP API server and business logic
- `main.go`: Application entry point, server setup
- `internal/blogger/`: Client for Blogger API (fetch posts, authenticate)
- `internal/config/`: YAML configuration parsing and validation
- `internal/httpapi/`: HTTP request handlers (schema, posts, XML generation endpoints)
- `internal/xmlgen/`: XML rendering engine (templates, field mapping)

**`frontend/`** – Vue 3 TypeScript single-page application
- `src/components/`: Reusable form components (input fields, groups, arrays, date picker)
- `src/composables/`: Vue composition API logic (API calls, form state management)
- `src/types/`: TypeScript interfaces (form schema, API responses)
- Built output goes to `web/static/` (Vite production build)

**`web/`** – Static assets directory (auto-generated by `npm run build`)
- Served by the Go backend at `/`
- Contains HTML entry point, compiled JS/CSS, favicon

**Configuration & Infrastructure**
- `config.yaml`: Application settings (blog ID, form schema, XML output template)
- `config.example.yaml`: Template with defaults (commit this, not `config.yaml`)
- `Makefile`: Build and development automation
- `Dockerfile`: Multi-stage build (frontend → static files, Go → binary, final distroless image)

## Development Workflow & Make Targets

Common workflow:

```bash
make setup          # First time only: install npm + Go dependencies
make dev            # Start both backend (watch) and frontend (HMR) concurrently
# ... edit code, auto-reloads triggered ...

```

Additional targets:

**Build & Production**
- `make build`: Compile Vue 3 app to `web/static/` + Go binary to `bin/blogger-xml-exporter`. Run this before deploying.
- `make build-docker`: Build production Docker image (multi-stage, ~50MB with distroless base)

**Code Quality**
- `make lint`: Run ESLint on frontend
- `make format`: Auto-format TypeScript/Vue code
- `make test`: Run Vitest unit tests (frontend)

**Cleanup**
- `make clean`: Remove build artifacts and binaries (`bin/`, `frontend/dist/`, etc.)

**Watch Targets (rarely used directly)**
- `make watch-backend`: Start Go server with file watching via `air` (auto-restart on code changes)
- `make watch-frontend`: Start Vite dev server with HMR (hot module reload)
- Both are started together by `make dev` with `-j` (parallel)

**Environment**
- `BLOGGER_API_KEY`: Required for backend; set before `make dev` (default: `dev-dummy-key` for testing)
- `CONFIG_PATH`: Config file location (default: `./config.yaml`)

## Development vs. Production Architecture

This is crucial to understand for both local development and deployment:

**Development Mode** (`make dev`)
- Frontend: Vite dev server runs on `http://localhost:5173` with HMR (hot module reload)
- Backend: Go server runs on `http://localhost:8080` with file watching
- Frontend → Backend communication: **Vite proxy** (`vite.config.ts`)
  - All API calls (`/api/*`) are proxied from `:5173 → :8080` transparently
  - This allows frontend and backend to be on different ports without CORS issues
  - Frontend code can call `fetch('/api/schema')` and it automatically routes to the backend
- **No static files needed** – frontend is served live from Vite, not from `web/static/`

**Production Mode** (after `make build`)
- Frontend: Compiled to `web/static/` (HTML, JS, CSS bundles)
- Backend: Single Go binary (`bin/blogger-xml-exporter`)
- Frontend → Backend communication: **Same origin** (no proxy needed)
  - Backend serves frontend static files at `/` (root path)
  - Both frontend and API are on the same hostname/port (e.g., `localhost:8080`)
  - Frontend code calling `fetch('/api/schema')` goes to the same server (no proxy, no CORS)
- When deployed: Run the single binary, which serves both UI and API

**Key Difference**: In dev, two separate servers talk via proxy. In prod, one server serves everything.

## Deployment

**This application is completely stateless**: no sessions, no persistent storage, no shared state between instances. All data flows through the Blogger API and HTTP requests. This makes it ideal for:
- Kubernetes deployments with multiple replicas
- Horizontal scaling and auto-scaling
- Rolling updates and blue-green deployments
- Containerized orchestration platforms

## Configuration

### Structure

`config.yaml` separates two independent concerns:

- **`form.items`**: Form layout tree (groups, fields, presets, rows, widths)
- **`xml.fields`**: XML output structure (element paths and value sources)

This separation avoids duplicating structure. A form field pulls from a post, but the XML output can reshape it completely. Static XML values (e.g., company metadata) don't need hidden form fields.

### Blogger Settings

```yaml
blogger:
  blogId: "1234567890123456789"
  maxResults: 20
```

### Server & Branding

```yaml
server:
  port: 8080

site:
  title: "XML Exporter"
  heading: "XML Exporter"

theme:
  primaryColor: "#2563eb"        # Brand color (hex)
  darkColor: "#1e40af"           # Dark variant (optional)
  lightColor: "#3b82f6"          # Light variant (optional)
```

Omit theme colors to use defaults (blue shades). Colors are applied as CSS variables at runtime.

### Assets

```yaml
assets:
  dir: "/app/assets"             # Optional external directory
  favicon: "favicon.ico"         # Filename within dir
  logo: "logo.png"               # Filename within dir
```

### Form Definition

```yaml
form:
  items:
    - type: group
      title: "Article"
      items:
        - type: text
          name: title
          label: "Title"
          source: title            # Pull from post.title
          row: 1
          width: 8
        - type: textarea
          name: description
          label: "Description"
          template: '{{ .title }} summary'  # Template-based
          row: 1
          width: 4
```

Field types: `text`, `textarea`, `date`, `list`, `select`, `array`, `group`.

Each field has exactly one of:
- `source: <dot.path>` - pull directly from post JSON
- `template: <go-template>` - computed value
- Neither - user fills manually

### XML Output

```yaml
xml:
  root: "item"
  filename: '{{ .title }}'  # Download filename (without .xml)
  namespaces:
    - name: ex
      value: "http://example.org/schema"
  fields:
    - xmlPath: "header/title"
      formField: title
    - xmlPath: "body/summary"
      template: '{{ source "content" | stripHTML }}'
    - xmlPath: "meta/generator"
      template: "Blogger XML Exporter"
```

Each XML field references a form field, a template, or is static.

### Presets (Quick-Fill Templates)

Groups can define reusable preset templates:

```yaml
- type: group
  title: "Author"
  presets:
    - label: "John Doe"
      values:
        first_name: "John"
        last_name: "Doe"
        email: "john@example.com"
```

See `config.example.yaml` for more detailed examples.

## API Endpoints

| Method | Path                    | Returns                           |
|--------|-------------------------|-----------------------------------|
| GET    | `/healthz`              | Health check                      |
| GET    | `/api/form-schema`      | Form layout + site config + theme |
| GET    | `/api/defaults`         | Pre-fill without post selection   |
| GET    | `/api/posts`            | List recent posts                 |
| GET    | `/api/posts?q=<search>` | Full-text search                  |
| GET    | `/api/posts/<id>`       | Single post + pre-filled form     |
| POST   | `/api/generate`         | Generate & download XML           |
